Data Flow — Contact Form Submission

End-to-end sequence from visitor form fill to Audiris inbox delivery.

1

Visitor fills the contact form

Fields: Full name · Company · Email · Service interest · Message (optional)
Hidden honeypot field present but invisible to humans.

2

Client-side validation (React Hook Form)

Blur-time and submit-time validation. Inline error messages per field. Submit button shows loading spinner, disabled to prevent double-submit.

3

POST /api/contact

JSON body: { name, company, email, service, message, honeypot }
Request sent to Next.js serverless function on Vercel.

4a

Bot detection — Honeypot check

If honeypot field is filled → return 200 OK silently. Submission discarded. Bot is not informed.

4b

Server-side validation

Required fields present · email matches RFC 5322 · field length limits.
Fail → 400 Bad Request with field-level error map.

5

Resend SDK — send email

From: noreply@audiris.com   To: hello@audiris.com
Subject: New enquiry from [name] — [company]
Body: all form fields in HTML email template.

6

Resend delivers via SMTP

Resend handles SPF/DKIM signing, retry logic, and delivery receipts. Target: < 5 minutes to Audiris inbox.

7

Response to browser

Success: 200 OK → form replaced by "Thank you! We'll be in touch within 1 business day."
Resend error: 500 → "Something went wrong. Please email us directly at hello@audiris.com"

Security Architecture

Defence-in-depth: 5 layered controls from transport to supply chain.

🔒 Layer 1 — Transport Security
  • HTTPS enforced on all routes (Vercel default — no configuration required)
  • HTTP → HTTPS permanent redirect (301)
  • TLS 1.2+ only · Auto-renewing Let's Encrypt certificates
  • HSTS: Strict-Transport-Security enforced via Vercel edge
🛡️ Layer 2 — HTTP Security Headers (next.config.js)
  • X-Content-Type-Options: nosniff
  • X-Frame-Options: SAMEORIGIN
  • X-XSS-Protection: 1; mode=block
  • Referrer-Policy: strict-origin-when-cross-origin
  • Permissions-Policy: camera=(), microphone=(), geolocation=()
  • Content-Security-Policy: restrictive; no inline scripts
⚡ Layer 3 — API Route Protection (/api/contact)
  • Honeypot field: bot submissions silently discarded (200 returned)
  • All required fields validated server-side (belt-and-suspenders)
  • Email format validated against RFC 5322
  • RESEND_API_KEY server-only env var — never reaches client bundle
  • No form data written to server logs (NFR-SEC08)
  • Rate limiting: Vercel platform-level DDoS protection
🇪🇺 Layer 4 — Privacy & GDPR Compliance
  • Cookie consent banner on first visit — analytics fires ONLY after explicit "Accept"
  • Consent stored in localStorage — banner not shown on return visits
  • No PII (names, emails) passed to analytics events (NFR-AN04)
  • Plausible analytics: cookieless by default, no cross-site tracking
  • Privacy Policy page: data collected, processors (Resend, analytics), opt-out instructions
📦 Layer 5 — Supply Chain Security
  • npm audit run pre-launch — no high/critical vulnerabilities permitted
  • Minimal dependency surface: Next.js · Tailwind · Framer Motion · Lucide · RHF · Resend SDK
  • Tree-shaken bundle — unused code never shipped to browser
  • .env.local gitignored — secrets never committed to repository
  • TypeScript strict mode — eliminates whole class of runtime errors at build time

Architecture Decision Records

Key decisions, their rationale, and the trade-offs accepted.

# Decision Rationale Trade-off
ADR-01 Next.js 14 App Router + SSG All pages pre-rendered at build time → no runtime server → SEO-perfect HTML → 99.9% availability via CDN alone Content updates require redeploy
ADR-02 Vercel hosting Zero-config for Next.js · atomic deploys · global edge CDN · preview environments per PR · free tier covers v1 traffic Vendor lock-in (mitigated by standard Next.js)
ADR-03 No CMS at launch Reduces complexity, cost, and surface area. Content in lib/content.ts. CMS added in v2.0 when content velocity justifies it Non-technical edits require developer
ADR-04 Resend for email delivery Simplest API · reliable deliverability · SPF/DKIM support · 3k/month free tier exceeds v1 needs Low risk — easy to swap if needed
ADR-05 Tailwind CSS Utility-first enforces design token consistency · eliminates unused CSS via PurgeCSS · fast iteration speed Verbose HTML class attributes
ADR-06 Framer Motion for animations Declarative whileInView scroll reveals · built-in prefers-reduced-motion support · tree-shakeable ~30KB bundle cost
ADR-07 React Hook Form Uncontrolled inputs = better performance · no schema library dependency at 9KB · native validation integration Minimal trade-off
ADR-08 Plausible (preferred) over GA4 Privacy-first by default · GDPR-compliant without complex cookie setup · no cross-site tracking · simpler consent flow Less granular data than GA4
ADR-09 Honeypot over CAPTCHA Zero friction for legitimate users · invisible implementation · sufficient for low-volume contact form · no 3rd-party JS Less effective against sophisticated bots
ADR-10 Single serverless function Contact form is the only stateful operation. Everything else is static. Simplest possible backend surface area. Smallest attack surface. No meaningful trade-off at v1 scale
ADR-11 Content in lib/content.ts (no JSX hardcoding) Separates concerns — content editors don't need JSX knowledge. Enables future CMS swap with minimal refactor. Indirection adds one lookup step
ADR-12 Custom Cookie Consent (no 3rd-party lib) Reduces bundle size · full control over UX · no SaaS cost · simple localStorage implementation sufficient for this use case Manual maintenance of consent logic
